# Codetta — ビジョン / スコープ

> **Codetta** = `code` + `coda` (音楽用語: 終結部)
> LLM ネイティブな軽量 DAW-like ツール。テキスト中心のプロジェクト形式と MCP 統合により、
> Claude や ChatGPT が直接「作曲する側」として参加できることを最大の差別化とする。

## ビジョン

LLM が**読み・書き・編集できる**音楽制作環境を作る。

既存 DAW (Cubase / Logic / FL 等) はバイナリ形式のプロジェクトファイル + マウス操作前提の UI を持つため、
LLM が「曲を作る」「部分修正する」「フィードバックに応じてリビジョンする」ことが構造的に困難である。
Codetta はこの制約を取り払い、**LLM が一級市民として作曲に参加できる最初の音楽ツール**を目指す。

## ターゲットユーザー

優先順位順:

1. **個人ゲーム開発者 / インディー作家** — BGM / SE が必要だが DTM スキルを習得する余裕がない
2. **LLM で作曲を試みる開発者** — Claude Code / Cursor / Claude Desktop ユーザー
3. **チップチューン・電子音楽愛好家** — 軽量・テキストベースのツールを好む層
4. **教育用途** — 音楽プログラミング教材としての利用

明示的に**ターゲット外**:
- プロの DTM ユーザー (Cubase / Pro Tools の置き換えではない)
- 録音中心のミュージシャン
- VST 大量利用が前提のユーザー

## 差別化 (3 本柱)

### 1. LLM ネイティブ (最大の差別化)

- MCP server を**コア提供物**として位置付ける (添え物ではない)
- プロジェクトファイル = 人間も LLM も読める JSON / DSL
- Claude が tools 経由で `create_song` / `add_notes` / `render_wav` を呼べる
- 「ダーク感をもう少し」のようなフィードバックで部分修正可能

競合状況: 既存 DAW で MCP 対応はほぼ無し (2026-05 時点)。**空き地ポジション**。

### 2. テキスト中心 (Git フレンドリー)

- プロジェクトファイルは JSON (バイナリブロブなし)
- `git diff` / `git merge` が意味を持つ
- 「曲のスニペット」をテキストで共有可能
- LLM プロンプトに直接コピペできる

### 3. 軽量・ポータブル

- 単一バイナリ配布 (Rust コア)
- 起動 1 秒以下を目標
- 大容量サンプルライブラリに依存しない (内蔵シンセ主体)
- 配布サイズ < 50MB を目標

## スコープ

### 含める (Phase 0-1)

| 機能 | 段階 |
|---|---|
| MIDI 打ち込み (内部 JSON 表現) | Phase 0 |
| 内蔵シンセ (saw/square/sin/triangle + ADSR + Filter) | Phase 0 |
| 内蔵ドラム音源 (TR-808/909 風) | Phase 0 |
| 簡易エフェクト (Reverb / Delay / Filter) | Phase 0 |
| WAV エクスポート (オフラインレンダリング) | Phase 0 |
| MIDI import / export | Phase 1 |
| MCP server (Claude 連携) | Phase 1 |
| GUI (ピアノロール、再生、微調整) | Phase 3 |

### 含めない (明示的に却下)

| 機能 | 却下理由 |
|---|---|
| **VST3 / AU プラグインホスト** | C++ 必須で開発期間 +1 年。LLM 連携と相性悪い (内部ブラックボックス) |
| **オーディオ録音** | 別問題、スコープ膨張要因 |
| **マスタリング機能** | プロ用途、ターゲット外 |
| **プラグイン SDK 提供** | 外部開発者向けは Phase 2 以降に判断 |
| **スコア (五線譜) UI** | 学習コストが高く、ターゲットユーザーに不要 |
| **大規模サンプルライブラリ** | 配布サイズ膨張、軽量原則と矛盾 |
| **クラウド同期 / 共同編集** | サーバ運用コスト、Phase 4 以降の課金機能候補 |

## ライセンス方針

- **Apache 2.0** (コア + CLI + MCP server + GUI)
- 理由:
  - MIT 並みの自由さ
  - 特許条項 (LLM 連携で将来トラブル余地)
  - 大手 OSS (Kubernetes / Spark / Android) と同じで信頼感あり
  - 将来の Open Core (Pro 機能を別ライセンス) 余地を残せる

将来の課金導線 (Phase 4 以降に判断):
- 案 A: 純 OSS のまま、寄付のみ (GitHub Sponsors)
- 案 B: コアは無料、Pro 機能 (クラウド同期 / プレミアム音色 / 共同編集) を別途有料
- 案 C: ホスト型 SaaS (Web 版 + クラウド) を有料、ローカル版は無料

現時点では決定しない。**Phase 0-3 はライセンスを純粋 OSS として運用**。

## ネーミングの由来

- **Codetta** = `code` + `coda` (音楽用語: 楽曲の終結部、添え書き)
- 「コード (プログラム) で曲を書く」「LLM が最後の仕上げ (coda) を担う」のダブルミーニング
- 商標衝突調査: 既存ソフトウェア名と被りなし (2026-05 時点、要再確認)
